# Architecture

This document gives you a big-picture tour of how a project generated by this template
is structured and why certain choices were made. It's aimed at developers joining the
project and wanting to get oriented quickly.

## Project Layout

The project uses a **`src`-layout**. Inside the `src/` directory you'll find two
top-level packages side-by-side:

```text
src/
├── <project_module>/   # The Django project - a single installable Python package
│   ├── __init__.py
│   ├── manage.py
│   ├── main/           # Settings, root URL conf, ASGI/WSGI, shared utilities
│   ├── accounts/       # Custom User model app
│   ├── static/
│   └── templates/
└── tests/              # The full test suite
    ├── conftest.py
    ├── factories/
    ├── fixtures/
    ├── functional/
    └── system/
```

The Django project is intentionally wrapped as a **single Python package**. This might
feel unusual if you're used to a flat collection of Django apps, but it has some real
benefits:

- **Cleaner imports** - everything lives under one namespace.
- **Simpler packaging** - `pyproject.toml` declares a single package with
  `packages = [{include = "<project_module>", from = "src"}]`, and building a wheel
  just works.
- **CLI entry points** - because the project is a proper package, we can define console
  scripts. The standard Django `manage.py` is exposed as a `manage` command via
  `[tool.poetry.scripts]` in `pyproject.toml`. Inside the virtualenv (or the Docker
  container) you just type `manage` instead of `python manage.py`. This is used
  throughout the entrypoint, invoke tasks, and day-to-day development.

### Django App Names Are Dotted

Because apps live inside the project package, their `AppConfig.name` must reflect the
full dotted path. For example, the accounts app declares:

```python
class AccountsConfig(AppConfig):
    name = "<project_module>.accounts"
```

This is a direct consequence of the package-based layout. You'll see the same pattern in
`INSTALLED_APPS` in `settings.py`. Any new apps you add should follow the same
convention.

## The `main` Package

The `main` package is the heart of the Django project. It contains:

- **`settings.py`** - project configuration, driven by environment variables.
- **`urls.py`** - the root URL configuration.
- **`asgi.py` / `wsgi.py`** - ASGI and WSGI application entry points.

Because it's the one package every other app already depends on (via settings and URL
routing), `main` is a convenient home for **reusable, cross-app code**. Think custom
middleware, shared template tags, or custom model field types. Rather than creating a
dedicated "utils" app, put things here when they genuinely serve the whole project.

`main` also ships with a couple of views out of the box:

- **`/humans.txt`** - renders a plain-text [humans.txt][humanstxt] page
  from a template.
- **`/release.json`** - returns JSON containing the git commit hash, timestamp, and
  revision count from the `main` branch. These values are baked into the Docker image at
  build time, so you can always tell exactly what code a running release maps to. See
  [Release Tracking](#release-tracking) below.
- **`/sentry-debug/`** - a `DEBUG`-only endpoint that raises an error, useful for
  verifying your Sentry integration is working.

## The `accounts` Package

This is a Django app providing a **custom `User` model** from the very first migration.
It deliberately does very little. It extends `AbstractUser` and overrides `__str__`.
The point is simply to have a custom user model in place from day one.

Why bother? Because migrating to a custom user model after you've already created
migrations is [notoriously painful][custom-user-docs]. Starting with one avoids that
problem entirely, and gives you a clean place to add custom fields, methods, or managers
later on.

## Configuration

Project configuration is driven by **environment variables**, read via
[django-environ][django-environ]. This keeps secrets out of
code and makes the app straightforward to configure across environments.

- For **local development**, copy `example.env` to `.env`. It's automatically loaded
  by `settings.py`.
- In **production**, environment variables are set directly (typically injected by your
  deployment platform or passed via `docker run --env-file`).

See [Environmental Variables][envvars] for the full list of supported variables and
their defaults.

## Release Tracking

Every Docker image has release metadata baked in at build time. The pipeline looks like
this:

1. **`tasks.py`** extracts git metadata. The commit hash, commit timestamp, and total
   commit count on the `main` branch.
2. These are passed to `docker build` as **build arguments** (`GIT_COMMIT_HASH`,
   `GIT_COMMIT_TIME`, `GIT_COMMIT_COUNT`).
3. The **Dockerfile** persists them as environment variables in the final image.
4. **`settings.py`** reads them via `django-environ`, making them available to
   application code.
5. The **`/release.json`** endpoint serves them as JSON, and they're attached to
   **Sentry releases** so errors can be correlated to exact commits.

The upshot: given any running instance, you can hit `/release.json` and know precisely
which commit it was built from.

## ASGI & Uvicorn

The project is **ASGI-first**. [Uvicorn][uvicorn] serves the
application directly.

The Docker image's `entrypoint.sh` orchestrates startup: it can run migrations, start
the server, open a shell, or pass arbitrary commands through to `manage`. See
[Docker Entry Point][entrypoint] for the full details.

## Tests

Tests live in `src/tests/` and are split into two broad categories:

### Functional Tests (`tests/functional/`)

These are the highest-level tests. They exercise the **publicly available interface** of
the project, typically by making real HTTP requests against the live Django test server,
driven by [Playwright][playwright]. If a user can do it in a
browser, a functional test can verify it.

### System Tests (`tests/system/`)

These cover **unit and integration tests** that exercise portions of the codebase in
isolation. The directory structure mirrors the app layout:

```text
tests/system/
├── accounts/
│   └── test_models.py
└── main/
    └── test_views.py
```

### Markers & Running Suites

You don't need to manually decorate tests with `@pytest.mark.functional` or
`@pytest.mark.system`, **`conftest.py` auto-applies markers** based on the directory a
test lives in. You can run a single suite with:

```shell
inv test --suite system     # just system tests
inv test --suite functional # just functional tests
```

### Factories & Fixtures

- **`tests/factories/`** - test data factories built with
  [factory_boy][factory-boy]. See
  [How to use factories][factories-howto] for conventions (one module per app,
  minimal defaults, no random/fuzzy fields).
- **`tests/fixtures/`** - shared [pytest fixtures][pytest-fixtures]
  used across the test suite.

## Documentation

The `docs/` folder holds **project-specific documentation** in Markdown, served locally
via [MkDocs][mkdocs] with the
[Material][mkdocs-material] theme. Run `inv run-docs` to
preview it.

The documentation is organised according to the
[Diátaxis][diataxis] framework, which splits docs into four categories based
on what the reader needs:

| Category      | Directory         | Purpose                              |
|---------------|-------------------|--------------------------------------|
| **Tutorials** | `docs/tutorials/` | Learning-oriented guided walkthroughs|
| **How-Tos**   | `docs/how-tos/`   | Task-oriented step-by-step recipes   |
| **Reference** | `docs/reference/` | Information-oriented facts and specs |
| **Decisions** | `docs/decisions/` | Architectural Decision Records (ADRs)|

Why Diátaxis? It prevents documentation from becoming an unstructured dump. Each
document has a clear purpose and audience, which makes it easier to find what you need
and keeps contributions focused. When writing new docs, pick the category that best fits
what the reader is trying to do.

ADRs are lightweight records of significant decisions. Why something was chosen, what
alternatives were considered, and what the implications are. See
[ADR 0000][adr-0000] for the format.

## Docker

The ultimate build artefact of the project is a **Docker image**. The multi-stage
`Dockerfile` produces a lean runtime image:

1. **Packager** - exports pinned dependencies and builds a Python wheel.
2. **Builder** - creates a virtualenv, installs dependencies, then installs the app
   wheel.
3. **Runtime** - copies the virtualenv into a minimal base image with only the necessary
   system libraries.

There is **no infrastructure or deployment tooling** included. The template stops at
producing the image. How and where you deploy it is up to you.

For local development, `docker-compose.yml` provides a **PostgreSQL** service only. The
Django application itself runs outside Docker during development (directly via
`manage runserver` or through invoke tasks).

## Local Development with Invoke

Day-to-day development workflows are defined as
[Invoke][invoke] tasks in `tasks.py`. Key tasks include:

| Task                   | What it does                              |
|------------------------|-------------------------------------------|
| `inv format`           | Formats code with Ruff and sorts imports  |
| `inv lint`             | Lints code with Ruff                      |
| `inv typing`           | Runs mypy via the daemon for speed        |
| `inv test`             | Runs the test suite with coverage         |
| `inv check`            | Runs typing, tests, lint, migration check |
| `inv check-migrations` | Verifies no migrations are missing        |
| `inv build-image`      | Builds the Docker image with git metadata |
| `inv build-docs`       | Builds the documentation site             |
| `inv run-docs`         | Serves the documentation locally          |

The guiding philosophy is: **CI should just call these same tasks.** Anything that runs
in your CI pipeline should be reproducible locally with a single `inv` command. This
keeps the feedback loop tight and avoids "works on my machine" surprises. `inv check` is
the go-to command for running the full QA suite before pushing.
